# Account Extractor - 账号提取测试 Skill

本 skill 用于从工程源码和配置文件中提取所有账号信息，采用分步执行、subagent 机制、即时落盘策略。

---

## 一、执行流程

**完整流程**：向用户询问/确认 → 撰写测试计划 → 再次用户确认 → 执行测试

### 1.1 信息收集阶段（必须执行）

**逐一询问用户以下问题**：

| 序号 | 问题 | 说明 |
|------|------|------|
| 1 | 检视根目录路径？ | 用户输入工程根目录（如 `D:\project`） |
| 2 | 测试结果存放路径？ | 默认 `<根目录>/TestPlan/AccountTest/`，用户可自定义 |
| 3 | 账号类型定义？ | 默认提供 7 类，用户可增删改 |
| 4 | 排除范围？ | 默认排除 `.md` 文档，用户可添加排除目录 |
| 5 | 文件类型范围？ | 默认 `.ets/.ts/.vue/.json/.json5/.js/.jsx/.tsx/.py/.java/.xml`，用户可调整 |

**默认账号类型**：

| 类型编号 | 类型名称 | 搜索关键词示例 |
|----------|----------|----------------|
| A1 | 硬编码用户名 | `'ApiAdmin'`, `'admin'`, `'root'`, `userName =` |
| A2 | 手机号 | `1[3-9]\d{9}`, `phone`, `138****` |
| A3 | 邹箱地址 | `@\w+\.com`, `@\w+\.cn`, `email` |
| A4 | SIP用户ID | `sip_user_id` |
| A5 | 云服务用户账号 | `AppUser`, `user`, `userId` |
| A6 | 凭证值中嵌入的用户名 | 从密码/Token值中提取用户名部分 |
| A7 | Mock/测试账号 | 测试代码中的假账号，需单独标记 |

---

### 1.2 撰写测试计划阶段

根据用户确认的信息，撰写测试计划文档，保存至 `<结果目录>/账号提取测试计划.md`。

**测试计划必须包含**：
1. 测试概述（账号定义、测试目的）
2. 测试范围界定（涉及目录、排除范围、文件类型）
3. 账号类型分类（用户提供或默认）
4. **测试执行原则**（强调分步执行、即时落盘、subagent 策略）
5. 测试步骤（按模块分组，具体搜索命令）
6. 输出要求（结果文件格式）

---

### 1.3 再次确认阶段

向用户展示测试计划摘要，询问：
- "测试计划已生成，存放于 `<路径>/账号提取测试计划.md`。请确认是否开始执行测试？"
- 如用户有修改意见，更新测试计划后再确认

---

### 1.4 执行测试阶段

**核心策略**：

| 策略 | 说明 |
|------|------|
| 分步执行 | 按文件逐一测试，不批量执行 |
| 即时落盘 | 每完成一个文件立即写入结果文件 |
| Subagent 机制 | 对 ≤1000 行的文件启动 explore subagent，完成后释放 |
| 结果追加 | 后续测试结果追加到已有结果文件 |

---

## 二、Subagent 使用规则

### 2.1 创建条件
- 文件行数 ≤ 1000 行：创建独立 subagent 检视
- 文件行数 > 1000 行：主 agent 直接检视（或分段处理）

### 2.2 Subagent Prompt 模板

```
请检视文件 `<文件路径>`，提取其中包含的所有账号信息。

**账号定义**（本次提取范围）：
- 硬编码用户名：如 `'ApiAdmin'`, `'admin'` 等实际赋值
- 手机号：用于身份识别的电话号码
- 邹箱地址：用于身份识别的电子邮件
- SIP用户ID：sip_user_id 字段的实际值
- 云服务用户账号：AppUser、user 等字段的实际值
- 凭证值中嵌入的用户名：如密码 `'Admin@12345'` 中包含用户名 `'Admin'`（仅提取用户名部分）

**不提取的内容**：
- 密码字段定义（password: string）
- API密钥（access_key, secret_key）
- Token相关定义
- 存储键名格式

**请返回**：
1. 文件行数
2. 发现的账号列表（格式：行号 | 账号内容 | 账号类型）
3. 若未发现账号，返回"无账号发现"

请读取文件并仔细检视每一行代码，不要遗漏任何账号信息。
```

### 2.3 释放时机
- Subagent 返回结果后立即释放（不保留 task_id）
- 将结果落盘后再启动下一个 subagent

## 三、结果落盘规则

### 3.1 结果文件路径
```
<结果目录>/账号提取结果.md
```

### 3.2 结果文件格式

```markdown
# 账号提取测试结果

> 测试时间: YYYY-MM-DD
> 测试范围: <用户指定的范围>
> 执行状态: 进行中/已完成

---

## Subagent 执行记录

| 序号 | 启动时间 | 目标文件/模块 | Subagent状态 | 发现账号数 | 释放状态 |
|------|----------|---------------|--------------|------------|----------|
| 1 | 进行中 | Xxx.ets | 运行中 | - | 待释放 |

---

## A1 类型：硬编码用户名

### [文件名] 检视结果

**文件路径**: `<完整路径>`
**文件行数**: N 行
**检视方式**: explore subagent (task_id)

| 行号 | 账号内容 | 说明 |
|------|----------|------|
| N | admin | `[业务代码]` / `[Mock测试数据]` / `[配置文件]` |

---

## A2 类型：手机号
...

## 统计汇总

（测试完成后添加）
```

### 3.3 落盘时机
- **每完成一个文件**：立即 edit 结果文件，追加该文件的检视结果
- **同时更新 Subagent 执行记录表**：标记状态为"已完成"、"已释放"

## 四、测试执行顺序建议

按以下顺序执行，确保高效：

1. **第一批**：核心配置文件（Constants、Config 等）
2. **第二批**：Handler/Controller 目录逐文件测试
3. **第三批**：Service/API 目录逐文件测试
4. **第四批**：Model/Bean 目录批量测试
5. **第五批**：Pages/Views 目录分批测试
6. **第六批**：前端 Stores/State 目录测试
7. **第七批**：配置文件（JSON/YAML/XML）测试
8. **第八批**：特殊模式全局搜索（手机号、邮箱正则）

---

## 五、账号来源标注规则

| 标注 | 适用场景 |
|------|----------|
| `[业务代码]` | 生产代码中的真实账号 |
| `[Mock测试数据]` | 测试/Mock 代码中的假账号 |
| `[配置文件]` | JSON/YAML/XML 配置中的账号 |
| `[注释说明]` | 注释中提到的账号说明 |
| `[UI提示文字]` | UI 字符串资源中的账号 |

---

## 六、注意事项

1. **聚焦账号信息**：仅提取用户名、手机号、邮箱等身份标识
2. **不提取凭证**：密码、API密钥、Token 不在提取范围（但可提取凭证值中嵌入的用户名）
3. **避免重复**：同一账号在不同位置出现时，标注所有位置
4. **区分字段定义与实际值**：
   - 字段定义（如 `userName: string`）不记录
   - 实际赋值（如 `userName = 'admin'`）记录为账号
5. **上下文管理**：使用 subagent + 即时释放策略，避免上下文溢出

---

## 七、测试终止条件

- 所有指定目录和文件类型均已测试完成
- 结果文件已完整记录所有发现的账号
- 用户指定的账号类型均已覆盖搜索
