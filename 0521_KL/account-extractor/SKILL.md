# Account Extractor - 账号提取测试 Skill

本 skill 用于从工程源码和配置文件中提取所有账号信息，采用分步执行、subagent 机制、即时落盘策略。

---

## 一、执行流程

**完整流程**：向用户询问/确认 → 撰写测试计划 → 再次用户确认 → 执行测试

### 1.1 信息收集阶段（必须执行，不可跳过）

**此阶段绝对不可跳过。** 在收集到所有必问信息之前，禁止开始测试。禁止自行推断或假设用户的答案。

**提问规范**：
- **逐一提问**：每次只问 1~2 个问题，等用户回答后再问下一批
- **不预设答案**：即使用户消息中包含部分信息，也必须确认后再使用
- **确认关键参数**：根目录和输出目录必须从用户获取，不可自行创建

**第 1 轮（必问）**：
> 1. 检视根目录路径？（如 `D:\project\src\`）
> 2. 测试结果存放路径？（默认 `<根目录>/TestPlan/AccountTest/`）

等待用户回复后，继续第 2 轮。

**第 2 轮（必问）**：
> 3. 账号类型是否使用默认 7 类（A1~A7），还是自定义？
> 4. 排除范围和文件类型？（默认排除 `.md`；默认文件类型 `.ets/.ts/.vue/.json/.json5/.js/.jsx/.tsx/.py/.java/.xml`）

等待用户回复后，口头总结关键参数，确认无误后进入测试计划阶段。 |

**默认账号类型**：

| 类型编号 | 类型名称 | 搜索关键词示例 |
|----------|----------|----------------|
| A1 | 硬编码用户名 | `'ApiAdmin'`, `'admin'`, `'root'`, `userName =` |
| A2 | 手机号 | `1[3-9]\d{9}`, `phone`, `138****` |
| A3 | 邹箱地址 | `@\w+\.com`, `@\w+\.cn`, `email` |
| A4 | SIP用户ID | `sip_user_id` |
| A5 | 云服务用户账号 | `AppUser`, `user`, `userId` |
| A6 | 凭证值中嵌入的用户名 | 从密码/Token值中提取用户名部分 |
| A7 | Mock/测试账号 | 测试代码中的假账号，需单独标记 |

---

### 1.2 撰写测试计划阶段

根据用户确认的信息，撰写测试计划文档，保存至 `<结果目录>/账号提取测试计划.md`。

**测试计划必须包含**：
1. 测试概述（账号定义、测试目的）
2. 测试范围界定（涉及目录、排除范围、文件类型）
3. 账号类型分类（用户提供或默认）
4. **测试执行原则**（强调分步执行、即时落盘、subagent 策略）
5. 测试步骤（按模块分组，具体搜索命令）
6. 输出要求（结果文件格式）

---

### 1.3 再次确认阶段

向用户展示测试计划摘要，询问：
- "测试计划已生成，存放于 `<路径>/账号提取测试计划.md`。请确认是否开始执行测试？"
- 如用户有修改意见，更新测试计划后再确认

---

### 1.4 执行测试阶段

**核心策略**：

| 策略 | 说明 |
|------|------|
| 分步执行 | 按文件逐一测试，不批量执行 |
| 即时落盘 | 每完成一个文件立即写入结果文件 |
| 过滤修复建议 | SubAgent 结果写入前，主 Agent 检查并删除修复性内容（"修复建议"、"应改为"等） |
| Subagent 机制 | 对 ≤1000 行文件启动 subagent，完成后释放 |
| 结果追加 | 后续测试结果追加到已有结果文件 |

---

## 二、Subagent 使用规则

### 2.1 创建条件
- 文件行数 ≤ 1000 行：创建独立 subagent 检视
- 文件行数 > 1000 行：主 agent 直接检视（或分段处理）

### 2.2 Subagent Prompt 模板

```
请检视文件 `<文件路径>`，提取其中包含的所有账号信息。

**账号定义**（本次提取范围）：
- A1 硬编码用户名：如 `'ApiAdmin'`, `'admin'`, `'root'` 等实际赋值的字符串
- A2 手机号：符合 `1[3-9]\d{9}` 格式的电话号码
- A3 邮箱地址：符合 `xxx@domain.com` 格式的电子邮件
- A4 SIP用户ID：sip_user_id 字段的实际赋值的值
- A5 云服务用户账号：AppUser、user、userId 等字段的实际赋值的值
- A6 凭证值中嵌入的用户名：如密码 `'Admin@12345'` 中提取用户名 `'Admin'`
- A7 Mock/测试账号：测试代码中的假账号

**不提取的内容**：
- 密码字段定义（password: string）— 仅类型声明，无实际值
- API密钥（access_key, secret_key）
- Token相关定义
- 存储键名格式
- 纯字段类型定义（如 `userName: string`、`admin: boolean`）— 不是实际赋值

**重要约束 — 必须遵守**：
- 仅记录发现，绝对不提供修复方案或修复建议
- 不要输出"修复建议"、"应改为"、"建议修改"等修复性内容

**误报排除检查 — 对每个发现必须执行**：
在标记账号之前，先检查是否属于以下模式。若符合，标记为"🔍 疑似误报"而非直接标记为账号：
- 字段类型定义中的名称（如 `userName: string`、`admin: boolean`）— 不是实际赋值
- example.com / test.com / domain.com / placeholder.com 等已知示例域的邮箱
- 全为同一数字或有明显规律的手机号（如 `13800000000`、`12345678901`、`11111111111`）
- 版本号格式（如 `v1.0.0`、`v2.1.3`）— 看起来像数字但不是手机号
- 变量名含 type/key/field/label 但没有实际赋值的

注意: URL路径(/admin)、文件路径(/root/)、注释中的内容不在误报排除范围，需正常提取。

**输出格式**：
若文件中有账号，按以下表格逐行输出：
| 行号 | 账号内容 | 账号类型 | 判断 | 来源标注 |
|------|---------|---------|------|---------|
判断使用: ✅ 确认 / 🔍 疑似误报
来源标注: [业务代码] / [Mock测试数据] / [配置文件] / [注释说明] / [UI提示文字]
若文件中无账号，输出: "无账号发现"
最后给出小结: 共发现 N 处，其中确认 M 处，疑似误报 K 处。

请读取文件并仔细检视每一行代码。
```

### 2.3 释放时机
- Subagent 返回结果后立即释放（不保留 task_id）
- 将结果落盘后再启动下一个 subagent

## 四、测试执行顺序（按风险优先级）

| 阶段 | 范围 | 优先级 | 理由 |
|------|------|--------|------|
| **阶段 1** | `Constants` / `Config` / 配置文件 | 最高 | 最可能包含硬编码用户名和默认账号 |
| **阶段 2** | `Handler` / `Controller` / `Service` / `API` | 高 | 业务逻辑层，常包含认证用户信息 |
| **阶段 3** | `Model` / `Bean` / `stores` | 中 | 数据模型中的默认值和字段值 |
| **阶段 4** | `Pages` / `Views` / `Components` | 低 | UI 层，账号信息相对少见 |
| **阶段 5** | 全文件正则搜索（手机号、邮箱） | 中 | 补充扫描，覆盖跨文件的模式匹配 |

---

## 五、账号来源标注规则

| 标注 | 适用场景 | 示例 |
|------|----------|------|
| `[业务代码]` | 生产代码中直接赋值的真实账号 | `const user = 'ApiAdmin'` |
| `[Mock测试数据]` | 测试/Mock 文件中的假账号 | `const testUser = 'admin'` |
| `[配置文件]` | JSON/YAML/XML/.env 等配置 | `"username": "root"` |
| `[注释说明]` | 注释中提到的账号 | `// default: admin` |
| `[UI提示文字]` | UI 字符串/提示文本中的账号 | `placeholder="Enter email"` |
| `[凭证嵌入]` | 从密码/Token 中提取的用户名部分 | `'Admin@12345'` → `'Admin'` |
| `[URL嵌入]` | URL 路径中嵌入的用户名 | `/api/user/admin/profile` |

## 六、注意事项

1. **聚焦账号信息**：仅提取用户名、手机号、邮箱等身份标识
2. **不提取凭证**：密码、API密钥、Token 不在提取范围（但可提取凭证值中嵌入的用户名）
3. **避免重复**：同一账号在不同位置出现时，标注所有位置
4. **区分字段定义与实际值**：
   - 字段定义（如 `userName: string`）不记录
   - 实际赋值（如 `userName = 'admin'`）记录为账号
5. **上下文管理**：使用 subagent + 即时释放策略，避免上下文溢出
6. **疑似误报不剔除**：误报排除清单仅用于标注（🔍 疑似误报），不得直接剔除不报告。所有疑似点必须呈现给用户，让用户做最终判断
7. **URL/路径/注释不排除**：出现在 URL 路径、文件系统路径、代码注释中的账号仍作为正常发现提取，不因位置特殊而降级为误报
8. **主 Agent 过滤职责**：接收 SubAgent 结果后，检查并删除修复建议段落后再写入结果文件

## 七、测试终止条件

- 所有指定目录和文件类型均已测试完成
- 结果文件已完整记录所有发现的账号
- 用户指定的账号类型均已覆盖搜索
