# 证书使用场景白盒测试执行器

对源码项目执行证书使用安全性白盒静态分析，采用 SubAgent 逐文件独立上下文测试机制，避免上下文溢出。

---

## 一、总体流程（四阶段）

```
阶段1: 信息收集     →  逐一询问用户确认参数
阶段2: 撰写测试计划  →  生成测试计划文档，写入文件
阶段3: 用户确认      →  用户审阅计划，回复"确认"或修改意见
阶段4: 执行测试     →  按计划逐文件 subagent 测试，即时落盘
阶段5: 汇总结论     →  统计汇总 + CLAUDE.md/项目规范对照检查
```

**关键约束**：
- 阶段1 不可跳过，必须从用户获取所有必问信息
- 阶段3 不可跳过，用户未确认前不得进入阶段4
- 阶段4 中 subagent 逐一创建（不可并行），每个 subagent 完成任务后主 agent 立即落盘

### 2.1 第 1 轮（必问）

> 1. 请提供需要扫描的**根目录路径**。例如：`D:\hwcam-main`、`/home/user/project`、`src/`

等待回复后口头确认路径可访问，然后继续第 2 轮。

### 2.2 第 2 轮（必问）

> 4. 是否有需要**排除**的目录或文件类型？（如 `node_modules/`、`.git/`、`*.md` 文档、测试文件等）

等待回复后继续第 3 轮。

### 2.3 第 3 轮（按需）

> 5. 是否有项目特定的**编码规范文档**（如 `CLAUDE.md`、`CODING_STANDARDS.md`）需要在测试时对照检查？
> 6. 是否有**已知的证书文件存放目录**需要重点关注？（如 `certs/`、`rawfile/certs/`、`resources/ssl/` 等）

### 2.4 信息收集完成确认


## 三、阶段2：撰写测试计划

### 3.1 代码结构探索

在撰写计划前，**必须**先用 Explore subagent 探索项目结构：

```
探索 {ROOT_DIR} 的目录结构：
1. 列出所有源码目录及文件数量
2. 搜索证书相关关键词：cert, certificate, p12, pfx, pem, ssl, tls, X509, HUKS,
   keystore, CA, publicKey, privateKey, certManager, certFramework, networkSecurity,
   crypto, encrypt, decrypt, sign, verify, fingerprint, TLS, HTTPS, remoteValidation
3. 列出所有配置文件路径
4. 检查是否存在实际证书文件（.pem, .p12, .pfx, .cer, .crt, .der, .p7b, .key）
```

### 3.2 测试计划文档结构

根据探索结果，生成测试计划写入 `{OUTPUT_DIR}/证书使用场景白盒测试计划.md`。

**计划必须包含的章节**：

```markdown
# 证书使用场景白盒测试计划

> 创建时间: YYYY-MM-DD
> 测试范围: {ROOT_DIR}
> 测试项目: 证书使用场景合规性分析
> 测试引擎: Claude Code (SubAgent 机制)

## 一、测试概述
- 测试目的：识别证书使用场景、解析证书字段、验证安全合规性
- 测试方法：逐文件 SubAgent 独立上下文静态分析
- 预期产出：测试结果报告 + 统计摘要

## 二、测试范围界定
- 纳入范围：列出所有源码目录 + 配置文件 + 实际证书文件
- 排除范围：node_modules/、.git/、.md 文档、测试代码（按用户要求）

## 三、测试执行原则
1. 每个文件启动一个独立 subagent，逐一执行，不可并行
2. 每完成一组/一个文件后立即追加结果落盘
3. Subagent 结果精简，仅返回发现摘要
4. 主 agent 不阅读源文件，只接收 subagent 结果并落盘
5. 仅记录发现，**禁止**提供修复方案或建议

## 四、测试步骤
按优先级分组：
- P0 核心安全模块（加密/证书管理/密钥存储文件）
- P1 网络层（HTTP 客户端、TLS 配置、请求发送）
- P2 桥接/中间件层（API 网关、路由分发、RPC 调用）
- P3 配置与资源文件（构建配置、模块配置、实际证书文件）
- P4 前端/UI 层（页面视图、组件、状态管理）
- P5 其余源码（业务逻辑、数据模型、工具类）
- P6 项目规范对照检查（全部测试完成后执行）

每个分组列出具体文件路径和测试关注点。

## 五、SubAgent 标准化测试指令
[包含 10 个分析维度的具体指令模板]

## 六、结果输出格式
[包含发现记录格式、无发现记录格式、统计摘要格式]

## 七、风险等级定义
[高/中/低/信息 四级的判定标准]

## 八、执行流程总览
[ASCII 流程图]
```

### 3.3 SubAgent 标准化测试指令（必须包含在计划中）

计划中必须包含以下供每个 subagent 使用的分析维度：

```
**分析维度**:
1. **证书加载**：是否存在证书文件的读取操作
2. **证书解析**：是否存在对证书字段（Subject, Issuer, Serial,
   NotBefore, NotAfter, PublicKey, SAN等）的解析操作
3. **证书验证**：是否存在证书校验代码（签名验证、证书链校验、
   有效期检查、吊销检查）。注意检查跳过验证的配置
4. **证书/密钥存储**：是否存在证书或私钥的存储操作
5. **密钥使用**：是否存在使用证书公钥/私钥进行加解密、签名验签
6. **TLS/HTTPS 配置**：TLS 版本配置、证书信任配置、caPath 等
7. **硬编码凭据**：硬编码的证书字符串、私钥、密钥材料、密码
8. **算法强度**：加密算法及其密钥长度是否符合安全要求
9. **有效期**：证书有效期检查相关代码
10. **权限声明**：证书相关的系统权限声明
```

## 五、阶段4：执行测试

### 5.1 初始化

1. 创建结果报告文件 `{OUTPUT_DIR}/证书使用场景测试结果.md`，写入头部：

```markdown
# 证书使用场景白盒测试结果
> 测试执行时间: YYYY-MM-DD
> 测试引擎: Claude Code SubAgent
> 总测试文件数: 待统计
> 有发现文件数: 待统计
> 无发现文件数: 待统计
---
## 测试发现明细
```

2. 按计划中的 P0→P1→P2→P3→P4→P5→P6 顺序执行

### 5.2 逐文件 SubAgent 测试流程

对每个文件执行以下循环：

```
主 Agent                              SubAgent
  │                                      │
  ├─→ 创建 subagent ──────────────────→ 阅读文件内容
  │                                      │ 按10个维度逐一分析
  │                                      │ 生成发现摘要
  │  ←──── 返回发现结果 ──────────────── │ 释放上下文
  │
  ├─→ 读取当前报告文件
  ├─→ 追加本文件测试结果段
  ├─→ 写入报告文件
  │
  └─→ 继续下一个文件...
```

**subagent prompt 模板**（发送给每个 subagent）：

```
你是一个白盒代码安全审计员。请阅读以下文件，分析其中与证书使用相关的代码。

文件路径: {FILE_PATH}

分析维度:
1. 证书加载：证书文件读取（.pem, .p12, .pfx, .cer, .crt, .der, .p7b 等）
2. 证书解析：证书字段解析（Subject, Issuer, Serial, NotBefore, NotAfter, PublicKey, SAN 等）
3. 证书验证：签名验证、证书链校验、有效期检查、吊销检查；特别注意跳过验证的配置
   （如 remoteValidation: 'skip', verify=False, check_ca:false, insecureSkipVerify 等）
4. 证书/密钥存储：HUKS, KeyStore, localStorage, fileSystem, Preferences 等
5. 密钥使用：公钥/私钥的加解密、签名、验签操作
6. TLS/HTTPS 配置：TLS 版本、caPath、clientCert、key、rejectUnauthorized 等
7. 硬编码凭据：证书字符串、私钥、密钥材料、密码
8. 算法强度：加密算法及密钥长度是否符合安全要求
9. 有效期：notBefore/notAfter 相关代码
10. 权限声明：证书相关的系统权限

输出要求：
- 如有发现，逐条输出：
  [发现#] | {维度} | {行号或key名} | {风险:高/中/低/信息} | {事实描述} → {风险原因}
- 如无发现，输出：[无发现] 该文件未涉及证书使用相关代码
- 每条发现必须包含行号（源码文件）或 key 名（配置文件）
- 每条发现必须用 "→" 连接"是什么"和"为什么是问题"
- 不要提供修复建议
- 不要输出完整代码块
```

### 5.3 增量落盘机制

- 每完成**一个优先级分组**（如 P0 全部 4 个文件），立即追加结果到报告文件
- 若某个分组文件数超过 20 个，可每 10 个文件追加一次
- 每次追加使用 Read→Edit→Write 流程，确保证不丢失已有内容
- "无发现"的文件也需要记录，格式为：`### [分组] 文件路径` + `[无发现] 简述`

### 5.4 分组测试要点

**P0 核心安全模块**：每个文件必须用独立 subagent 逐行审阅，不可批量。

**P3 配置与资源文件**：
- 对于 .properties / .env / .yaml 等配置文件，逐键值对检查
- 对于实际证书文件（.pem/.p12 等），尝试解析其 Subject、Issuer、Serial、NotBefore、NotAfter、SignatureAlgorithm、KeySize
- **无论证书目录是否存在，都必须在报告中记录检查结果**

**P4 前端/UI 层**（如适用）：
- 调整分析维度：关注是否在前端存储证书、是否定义证书字段模型、是否绕过 Bridge 直接调用原生证书 API
- 搜索关键词：cert, certificate, p12, pem, ssl, tls, publicKey, privateKey, crypto, encrypt, decrypt

**P6 规范对照检查**：
- 由主 agent 执行，不通过 subagent
- 回顾全部测试发现，逐条对照项目规范文档
- 输出格式：表格列出每项规范的符合/偏离/需人工确认状态

## 六、结果格式规范（必须遵守）

### 6.1 每条发现的记录格式

```markdown
### [{分组}] {文件路径}

| # | 维度 | 行号 | 风险 | 描述 |
|---|------|------|------|------|
| 1 | 证书验证 | L121 | 高 | remoteValidation: 'skip' 跳过服务端证书验证 → 中间人攻击可窃取或篡改 HTTPS 通信数据 |
| 2 | 硬编码凭据 | L13 | 高 | 硬编码密钥材料 LOCAL_KEY_PART = '541b3502...' → 攻击者反编译即可获取密钥分片，参与根密钥派生 |
```

### 6.2 格式硬性要求

1. **行号必填**：源码文件标注 `L{行号}`，配置文件标注 key 名（如 `storePassword`、`ENCRYPT_WORK_KEY`），目录检查标注引用方文件:行号
2. **因果必写**：每条描述的格式为 `{客观事实} → {安全影响/风险原因}`
3. **风险自洽**："高"级发现的原因必须说明直接安全影响；"低"级发现的原因必须说明缓解因素
4. **零修复建议**：全文（含总结章节）禁止出现"建议"、"应立即"、"需要修复"、"推荐"等修复指导表述

### 6.3 统计摘要格式

```markdown
## 统计摘要

| 风险等级 | 发现数量 |
|---------|---------|
| 极高风险 | N |
| 高风险 | N |
| 中风险 | N |
| 低风险 | N |
| 信息 | N |

### 按维度分布
| 维度 | 发现数 |
|------|--------|
| 证书加载 | N |
| ...

### 测试覆盖率统计
| 组别 | 测试文件数 | 有发现文件数 | 无发现文件数 |
|------|-----------|-------------|-------------|
| ...

## 测试结论
[按风险等级分类陈述事实，不提供修复方案]
```

---

## 七、风险等级定义

| 等级 | 定义 | 典型场景 |
|------|------|---------|
| **极高** | 凭据明文存储、密钥材料可被直接提取还原 | .properties 明文根密钥、硬编码私钥 |
| **高** | 直接影响安全性，可能导致中间人攻击、密钥泄露 | `skip` 证书验证、禁用 CA 检查、硬编码密码 |
| **中** | 增加攻击面或存在配置缺陷，有条件地增加风险 | 证书白名单缺失、证书文件缺失、路径暴露 |
| **低** | 代码有缓解因素，实际利用难度高 | 内存驻留密钥（需 root 权限）、受限环境使用 |
| **信息** | 非安全缺陷，值得关注或标记供人工判断 | 证书相关数据模型定义、算法使用记录 |

## 八、语言/框架自适应

根据用户在阶段1提供的语言/框架，调整 subagent 的关键词搜索策略：

| 语言/框架 | 证书验证跳过模式 | 证书加载模式 | 密钥存储模式 |
|-----------|----------------|-------------|-------------|
| ArkTS | `remoteValidation: 'skip'` | `getRawFileContent()` | `HUKS`, `certManager` |
| Java/Android | `verifyFalse`, `trustAllCerts` | `KeyStore.load()`, `CertificateFactory` | `AndroidKeyStore`, `KeyStore` |
| Python | `verify=False`, `check_hostname=False` | `ssl.SSLContext.load_cert_chain()` | `keyring`, `.env`, `os.environ` |
| Go | `InsecureSkipVerify: true` | `tls.LoadX509KeyPair()` | `os.ReadFile()`, `embed.FS` |
| JavaScript/TS | `rejectUnauthorized: false` | `fs.readFileSync('*.pem')` | `localStorage`, `.env` |
| Rust | `danger_accept_invalid_certs()` | `std::fs::read("cert.pem")` | `dotenv!()`, `env!()` |
| C#/.NET | `ServerCertificateValidationCallback` 返回 true | `X509Certificate2` | `SecureString`, `app.config` |

## 九、注意事项

1. **SubAgent 不可并行**：即使多个文件可同时处理，也必须逐一启动 subagent
2. **排除 .md 文件**：所有 Markdown 文档不纳入测试范围
3. **配置文件不要遗漏**：构建配置（build-profile.json5, pom.xml, Cargo.toml, go.mod）、环境配置（.env, .properties, appsettings.json）中的证书/密钥引用
4. **目录存在性必须记录**：代码引用的证书目录无论是否存在，报告中必须有明确记录
5. **上一次执行的经验教训**（记录以供后续执行参考）：
   - 测试计划中 P4（前端层）的目录必须全部覆盖，不可只测部分
   - 统计数字要和实际发现一致
   - "无发现"的文件也要记录，确保覆盖清单可核对
