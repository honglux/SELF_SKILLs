# 硬编码密码密钥安全测试执行器

本 skill 用于执行密码密钥硬编码问题的白盒安全测试，采用追溯分析法定位硬编码风险点。

## 二、测试执行原则

### 2.1 分步落盘原则

**核心原则**：将测试细分为小模块，完成一个模块后立即落盘，再进行下一个，结果在文件中追加。

执行流程：
```
获取文件列表 → 分组（按目录/文件类型）→ 测试第1组 → 落盘 → 测试第2组 → 追加落盘 → ... → 完成
```

**为什么这样做**：代码量庞大时上下文会溢出，分步落盘可保留所有发现，避免中途丢失。

### 2.2 追溯分析法

**核心方法**：从密码/密钥使用点反向追溯数据来源，判断是否为硬编码字符串。

判断流程：
```
发现疑似点 → 搜索变量定义位置 → 判断数据来源 → 结论
```

数据来源分类：
| 来源类型 | 判断结果 |
|---------|---------|
| 字符串字面量直接赋值 | **硬编码**（高风险） |
| 从配置文件读取 | 检查配置文件是否加密 |
| 从用户输入获取 | 非硬编码（安全） |
| 从安全存储/加密存储获取 | 非硬编码（安全） |
| 参数传递 | 追溯参数来源 |
| 动态生成（随机/HUKS等） | 非硬编码（安全） |

### 2.3 仅记录发现原则

只记录问题位置和内容，**不提供修复方案**，不执行修复操作。

## 三、测试方法详解

### 3.1 关键词搜索定位

搜索以下关键词定位疑似点：

**密码相关关键词**：
```
password | passwd | pwd | secret | credential | token | auth | key
```

**硬编码密码特征**：
```
Admin@123 | admin123 | password123 | root123 | test123
['"][a-fA-F0-9]{32}['"]  # 32位十六进制（128位密钥）
['"][a-fA-F0-9]{64}['"]  # 64位十六进制（256位密钥）
['"][a-fA-F0-9]{128,}['"]  # 128位以上（可能是加密密文或长密钥）
```

**注意区分**：
| 十六进制长度 | 可能含义 | 追溯重点 |
|------------|---------|---------|
| 32 位 (64 字符) | 密钥片段或 Access Key | 检查字段名是否含 `_encrypted` 或 `ak` |
| 64 位以上 | 加密密文或长密钥 | 检查是否为已加密数据，追溯解密代码 |
| 128 位以上 | 可能是完整加密密文 | 通常为加密存储，需确认加密方法 |

**配置相关关键词**：
```
ak | sk | access_key | secret_key | api_key | private_key
```

### 3.2 配置文件加密判断

对于配置文件中的敏感字段，判断方法：

**判断规则**：
| 特征 | 结论 |
|------|------|
| 字段名含 `_encrypted` 后缀 | 已加密（无风险） |
| 值为长十六进制（64位以上）且字段名暗示加密 | 可能已加密，需追溯代码确认 |
| 值为明文格式（如可见字符密码） | **硬编码**（高风险） |
| 值为短十六进制（32位）且字段名含"key" | 疑似明文密钥（需追溯） |

**示例**：
```
sk_encrypted_prod=4b728d18ddd4...  → 已加密（无风险）
OPTICAL_ROOT_KEY=bbce4a4f09ac...   → 疑似明文密钥（需追溯）
password=admin123                   → 硬编码密码（高风险）
```

### 3.3 Mock/调试代码区分

判断是否为调试代码：

| 特征 | 结论 |
|------|------|
| 文件名含 `test`、`mock`、`debug` | 调试代码 |
| 代码中有 `DEBUG ONLY`、`WARNING: mock` 注释 | 调试代码 |
| 使用 localStorage 存储密码（仅浏览器调试） | 调试代码 |
| 硬编码密码如 `Admin@12345` 用于模拟验证 | 调试代码（低风险，需标注） |

### 3.4 默认用户名判断

硬编码默认用户名（如 `admin`、`ApiAdmin`）通常是设备厂商标准做法：

| 情况 | 风险等级 |
|------|---------|
| 仅用户名硬编码，密码由用户输入 | 低风险 |
| 用户名 + 密码同时硬编码 | 高风险 |

### 3.5 使用独立 Agent 深度追溯（适用于中等文件）

对于**文件不太大（建议 500 行以内）**且可能包含敏感操作的情况，可使用**独立 Agent（独立上下文）**进行深度追溯分析。

**适用场景**：
- 文件涉及加密/解密操作（如 `encrypt`、`decrypt`、`Crypto` 类名）
- 文件涉及登录鉴权操作（如 `login`、`auth`、`DigestAuth`）
- 文件涉及密钥管理（如 `KeyManager`、`Keystore`）
- 关键词搜索发现多个疑似点，需要系统性追溯

**执行方法**（使用 Task 工具创建 subagent）：

```
使用 task 工具，subagent_type 选择：
- "general"：用于复杂追溯分析，执行多步骤任务
- "explore"：用于快速定位文件中的关键操作点
```

**Subagent 任务提示模板**：

```
你是一个硬编码密码密钥安全测试的追溯分析 Agent。请阅读以下文件，检查是否存在密码/密钥硬编码问题。

文件路径: [完整文件路径]

任务要求:
1. 阅读整个文件内容
2. 检查是否有以下操作：
   - 加密/解密操作（encrypt、decrypt、crypto、AES、RSA等）
   - 登录鉴权操作（login、auth、authenticate、Digest等）
   - 密钥管理操作（key、keystore、secret等）
   - 密码存储/读取操作（password、pwd、save、get等）
3. 对于每个发现的使用点，追溯数据来源：
   - 字符串字面量直接赋值 → 硬编码（记录位置和内容）
   - 参数传递 → 追溯参数来源（记录追溯路径）
   - 配置文件读取 → 检查配置文件内容（记录配置文件路径）
   - 加密存储 → 确认是否安全（记录加密方法）
4. 输出格式：
   - 发现列表（每个包含：行号、代码片段、数据来源追溯结果、风险判断）
   - 总结：是否发现硬编码问题

请返回完整的追溯分析结果。
```

**主 Agent 执行流程**：

1. 判断文件大小（通过读取文件行数或预估）
2. 如果文件 ≤ 500 行且符合适用场景，启动 subagent
3. **等待 subagent 完成**（Task 工具会返回结果）
4. 接收 subagent 返回的分析结果
5. 根据结果，**立即落盘**到测试结果文件
6. 继续下一个文件测试

**为什么使用独立 Agent**：
- 独立上下文，不受主 Agent 上下文限制
- 可以完整阅读文件，系统性分析
- 追溯链路更完整，不易遗漏
- 结果交付给主 Agent 后立即落盘，避免丢失

## 四、执行流程

### Step 1: 动态获取文件列表

根据用户提供的目录，动态获取所有文件：

```
获取目录下所有文件 → 过滤排除项 → 得到待扫描文件列表
```

**不要假设文件后缀**：使用 glob 或文件遍历获取实际存在的文件类型。

### Step 2: 文件分组

将文件按类型/目录分组，便于分步执行：

```
配置文件组 → 源码核心组（security/network/service等）→ 源码其他组 → Vue 组件组 → Vue 视图组 → 全量扫描组
```

分组原则：先扫描高风险区域（配置文件、加密模块、网络模块），再扫描其他区域。

**Vue 项目特殊分组**：
- 若存在 `.vue` 文件，需单独分组：
  - `components/` 目录 → Vue 组件组
  - `views/` 目录 → Vue 视图组
  - 其他 `.vue` 文件 → Vue 其他组

### Step 3: 分组测试执行

**每个文件执行流程**：

**方案 A：关键词搜索 + 追溯（适用于大文件或快速扫描）**
1. 搜索关键词定位疑似点
2. 逐一追溯每个疑似点的数据来源
3. 判断是否硬编码
4. 记录发现到结果文件

**方案 B：独立 Agent 深度追溯（适用于中等文件，详见 3.5）**
1. 判断文件大小和类型
2. 使用 Task 工具创建 subagent（subagent_type: "general"）
3. 等待 subagent 完成分析并返回结果
4. 接收结果，整理格式
5. **立即落盘**到结果文件

**判断使用哪种方案**：

| 文件情况 | 推荐方案 |
|---------|---------|
| 文件 > 500 行 | 方案 A（关键词搜索） |
| 文件 ≤ 500 行，含加密/鉴权操作 | 方案 B（独立 Agent） |
| 文件为配置文件 | 直接读取判断 |
| 文件为测试/Mock 文件 | 方案 A，标注调试用途 |

## 五、疑似点追溯检查清单

对于每个疑似点，执行以下追溯检查：

### 5.1 变量定义追溯

```
1. 找到变量首次赋值位置
2. 判断赋值来源：
   - 字符串字面量 → 硬编码
   - 函数调用 → 追溯函数返回值来源
   - 参数传递 → 追溯参数来源
   - 配置文件读取 → 检查配置文件内容
```

### 5.2 配置文件内容追溯

```
1. 定位配置文件
2. 检查字段名是否含加密标识
3. 检查字段值格式（明文/密文）
4. 追溯代码中是否对字段值进行解密操作
```

### 5.3 函数调用追溯

```
1. 找到函数定义
2. 检查函数返回值来源
3. 检查函数是否从安全存储获取数据
4. 检查函数是否进行解密操作
```

### 5.4 加密存储确认

```
1. 检查存储方法名是否含 encrypt/decrypt
2. 检查是否使用加密 API（如 CryptoUtils、HUKS）
3. 检查存储内容是否为密文格式
```

## 六、风险等级判断标准

| 问题类型 | 风险等级 | 判断依据 |
|---------|---------|---------|
| 源码中明文密码字符串 | 高风险 | 直接在代码中出现完整密码 |
| 配置文件明文密钥 | 高风险 | 配置文件中密钥未加密 |
| 密钥片段组合风险 | 高风险 | 多处密钥片段组合可还原完整密钥 |
| 配置文件疑似密钥 | 中风险 | 十六进制值，需追溯确认 |
| Mock调试密码 | 低风险 | 明确标注调试用途，不进入生产 |
| 默认用户名硬编码 | 低风险 | 仅有用户名，密码由用户输入 |
| 密钥别名常量 | 无风险 | 别名本身非敏感，实际密钥由安全机制管理 |
| 配置文件已加密密钥 | 无风险 | 字段含加密标识，值为密文 |

## 八、执行注意事项

1. **不跳过疑似点**：每个关键词匹配都应追溯，不能仅凭字段名判断
2. **区分调试代码**：Mock/调试代码需标注但不计入生产风险
3. **组合风险关注**：多处密钥片段可能组合还原完整密钥，需关联分析
4. **定期回顾测试方法**：执行过程中隔一段时间回顾此 skill，确保方法不偏离
5. **避免上下文溢出**：严格执行分步落盘，不等待全部完成再写入
6. **Subagent 串行执行**：启动一个 subagent → 等待返回 → 落盘 → 再启动下一个，避免并行导致结果混乱
7. **Subagent 结果验证**：接收 subagent 结果后，检查是否包含必要的追溯信息（行号、来源、风险判断）
8. **文件大小判断**：超过 500 行的文件不建议使用 subagent 全量阅读，改用关键词搜索方法

## 九、常见问题判断示例

### 示例 1：源码硬编码密钥

```typescript
const LOCAL_KEY_PART = '541b3502f1baf0cc6ce0673513d76fa3';
```
**判断**：32位十六进制字符串直接赋值 → 硬编码密钥（高风险）

### 示例 2：配置文件密钥

```properties
OPTICAL_ROOT_KEY=bbce4a4f09acdd05797d8af5cbc007e5
```
**判断**：字段名含"ROOT KEY"（敏感），无 `_encrypted` 后缀，32位十六进制 → 疑似明文密钥（需追溯代码如何使用）

### 示例 3：配置文件已加密密钥

```properties
sk_encrypted_prod=4b728d18ddd44800b546122ee6d36c8f...
```
**判断**：字段名含 `_encrypted`，长十六进制密文 → 已加密（无风险）

### 示例 4：密码参数传递

```typescript
async login(username: string, password: string) {
  // password 来自参数
}
```
**判断**：password 为参数 → 追溯调用方传入的值来源 → 若来自用户输入则安全

### 示例 5：加密存储

```typescript
const encrypted = await CryptoUtils.encryptWithAes(password);
await storage.set('device_pwd', encrypted);
```
**判断**：存储前加密 → 非硬编码（安全）

### 示例 6：Mock 调试密码

```typescript
// DEBUG ONLY: Mock verification
if (password === 'Admin@12345') {
  // 仅浏览器调试
}
```
**判断**：含 DEBUG 注释 → Mock 调试代码（低风险，需标注）

### 示例 7：使用独立 Agent 深度追溯

**场景**：文件 `CryptoUtils.ets` 约 250 行，含加密操作。

**执行步骤**：
1. 判断文件大小：250 行 ≤ 500 行，符合使用 subagent 条件
2. 启动 subagent（subagent_type: "general"）
3. 提示内容：
   ```
   文件路径: entry/src/main/ets/security/CryptoUtils.ets
   
   任务: 阅读文件，检查加密相关代码是否有硬编码密钥。
   追溯每个密钥使用点的数据来源，判断是否硬编码。
   返回发现列表（行号、代码片段、来源追溯、风险判断）。
   ```
4. 等待 subagent 返回结果
5. 收到结果：发现第 13 行 `LOCAL_KEY_PART = '541b3502...'` 为硬编码密钥
6. **立即落盘**到测试结果文件
7. 继续下一个文件

**Subagent 返回结果示例**：
```
发现 1:
- 行号: 13
- 代码: const LOCAL_KEY_PART = '541b3502f1baf0cc6ce0673513d76fa3';
- 来源: 字符串字面量直接赋值
- 追溯: 无外部来源，直接硬编码
- 风险: 高风险（32位十六进制密钥片段）

发现 2:
- 行号: 24, 65
```
