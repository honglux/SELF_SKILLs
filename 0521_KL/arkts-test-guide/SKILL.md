# ArkTS 测试指导文档生成 Skill

本 skill 用于创建 ArkTS 模块（`entry/src/main/ets`）的测试指导文档，指导后续 agent 按步骤执行测试。

---

## 一、工作流程

### 1.1 信息收集阶段

在撰写测试指导文档前，先向用户确认以下关键信息：

**必问问题**（逐一询问，不要一次性抛出）：

1. **测试项目**：本次测试的具体项目是什么？（如：命令注入、内存泄漏、空指针、线程安全等）
2. **测试范围**：是全量测试还是聚焦特定目录？（如：仅测试 jsbridge/ 目录）
3. **输出路径**：测试指导文档存放位置？
4. **已有文档**：是否有现有的测试计划或之前的测试结果需要参考？

**可选确认**（根据情况询问）：

- 是否需要参考特定的编码规范文件？
- 是否有特殊关注点或排除项？

### 1.2 文档生成阶段

根据用户确认的信息，生成测试指导文档。

---

## 二、测试指导文档结构

生成的文档必须包含以下章节：

### 2.1 文档头部

```markdown
# [测试项目名称] 测试指导文档

> 创建时间: YYYY-MM-DD
> 测试范围: entry/src/main/ets
> 测试项目: [具体测试项目]
```

### 2.2 测试概述

简要说明：
- 测试目的
- 测试方法概述（泛化描述，不绑定具体测试项目的具体方法）
- 预期产出

### 2.3 测试范围界定

明确列出：
- 测试涉及的目录/文件范围
- 排除范围（如有）

### 2.4 测试执行原则

**必须包含以下原则**：

1. **分步执行**：按文件或小模块逐一测试，不批量执行
2. **即时落盘**：每完成一个文件/模块测试后，立即将结果写入文件，不等待全部完成
3. **结果追加**：后续测试结果追加到已有结果文件中
4. **标注位置**：测试发现需标注具体文件路径和代码行号
5. **仅记录发现**：记录测试发现，不提供修复方案

### 2.5 测试步骤（核心章节）

按文件/模块列出测试步骤：

```markdown
## [模块/文件名] 测试步骤

### 文件信息
- 文件路径: entry/src/main/ets/xxx/Xxx.ets
- 文件功能: [简要描述]

### 测试要点
1. [要点1 - 泛化描述]
2. [要点2 - 泛化描述]

### 搜索命令
[具体的搜索命令，用于定位风险点]

### 结果输出要求
- 标注发现位置（文件:行号）
- 记录发现内容
- 不提供修复建议
```

### 2.6 输出文件说明

明确告知 agent：
- 测试结果文件路径
- 文件格式（Markdown）
- 结果追加方式

## 三、泛化原则

### 3.1 测试方法泛化

测试指导文档的测试方法描述必须是**泛化的**，不绑定具体测试项目的具体技术细节。

**错误示例**（过于具体）：
```markdown
### 命令注入测试要点
1. 搜索 childProcess.spawn 调用
2. 检查参数是否来自用户输入
3. 验证是否使用参数分离模式
```

**正确示例**（泛化）：
```markdown
### API 使用审计要点
1. 识别可能存在风险的关键 API 调用
2. 追溯数据来源（用户输入、设备上报、配置文件等）
3. 验证是否有输入验证或防护措施
```

### 3.2 具体搜索命令

虽然测试方法泛化，但搜索命令需要**具体**，帮助 agent 定位代码：

```markdown
### 搜索命令
搜索关键 API 或风险模式：
Get-ChildItem -Recurse -Filter "*.ets" | Select-String -Pattern "[具体关键词]"
```

## 四、提问规范

### 4.1 提问原则

- **简洁**：每次提问不超过 3 个问题
- **必要**：只问不清楚或用户可能遗漏的信息
- **顺序**：按信息收集顺序逐一询问

### 4.2 典型问题模板

| 问题 | 适用场景 |
|------|---------|
| "本次测试的具体测试项目是什么？" | 用户未明确说明测试项目 |
| "是否全量测试，还是聚焦特定目录？" | 用户未界定测试范围 |
| "测试指导文档存放在哪个目录？" | 用户未指定输出路径 |
| "是否有现有测试计划需要参考？" | 可能存在历史文档 |

---

## 五、ArkTS 模块约定

### 5.1 测试范围限定

**仅测试 `entry/src/main/ets` 目录**，不涉及其他模块。

### 5.2 目录结构探索

测试前需先探索 `entry/src/main/ets` 目录的实际结构，不要依赖固定目录假设。

**探索方法**：
```powershell
# 列出所有 ETS 文件
Get-ChildItem -LiteralPath "entry/src/main/ets" -Recurse -Filter "*.ets" | Select-Object FullName

# 或列出目录层级
Get-ChildItem -LiteralPath "entry/src/main/ets" -Directory -Recurse | Select-Object FullName
```

**注意**：
- 目录结构可能随项目版本变化
- 根据实际结构调整测试步骤的组织方式
- 建议按功能模块（如 jsbridge、service、network 等）分组测试

## 七、注意事项

1. **不执行测试**：本 skill 仅生成测试指导文档，不执行实际测试
2. **不关联具体测试项目方法**：测试方法描述必须泛化，用户会后续提供具体测试项目
3. **强调分步落盘**：确保 agent 按文件/模块分步测试并即时记录
4. **范围限定 ArkTS**：所有测试范围仅限 entry/src/main/ets 目录
