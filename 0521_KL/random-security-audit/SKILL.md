# 随机数安全审计 Skill

## 概述

本 Skill 对源代码项目执行密码学安全随机数使用合规性白盒审计。核心关注：**安全敏感场景（加密密钥、IV、Salt、Token、认证 nonce 等）是否使用了不安全的随机数函数（Math.random、Date.now、自定义 PRNG 等）。**

## 执行流程总览

```
Phase 0: 信息收集（向用户确认）
    ↓
Phase 1: 撰写测试计划 → 用户确认
    ↓
Phase 2: 按阶段执行审计（文件级 SubAgent）
    ↓
Phase 3: 最终汇总报告
```

**核心原则**：
- 每完成一个文件的审计立即追加写入结果文件，不在内存中累积
- 每个源文件使用独立 SubAgent（独立上下文），完成后释放
- 不审计 .md 文档文件
- 仅记录发现，不提供修复方案

---

## Phase 0: 信息收集

**此阶段绝对不可跳过。** 在收集到所有必问信息之前，禁止进入 Phase 1。禁止自行推断或假设用户的答案。禁止使用"我将假设..."的表述。

### 提问规范

- **逐一提问**：每次只问 1~2 个问题，等用户回答后再问下一批
- **不预设答案**：即使用户消息中包含部分信息（如目录路径），也必须确认后再使用
- **确认关键参数**：根目录和输出目录必须从用户获取，不可自行创建

### 必问问题（分 3 轮）

**第 1 轮**：
1. **审计根目录**：需要审计的源代码根目录是哪个？（如 `D:\project\src\`）
2. **输出目录**：审计结果文件存放在哪里？（如 `D:\project\TestPlan\RandomTest\`）

**第 2 轮**：
3. **项目语言/平台**：项目中使用了哪些编程语言或框架？（如 ArkTS + Vue.js、Java + Kotlin、Python、Go、TypeScript 等）
4. **重点关注区域**：有没有特别需要关注的子目录或模块？（如 `security/`、`network/`、`crypto/`）若用户无特殊要求，默认全量审计。

**第 3 轮**（可选确认）：
5. 是否有现成的随机数函数参考文档？（如 RandomFunctionsReport.md）
6. 是否有编码规范文件对随机数使用有要求？是否需要排除特定目录？

全部确认后向用户口头总结关键参数，再进入 Phase 1。

---

## Phase 1: 撰写测试计划

根据 Phase 0 收集的信息，撰写一份测试计划文档，存为 `[输出目录]/RandomSecurityAuditPlan.md`。

### 测试计划必须包含

1. **测试概述**：审计目的、方法（白盒静态分析）、预期产出
2. **测试范围**：包含的目录和文件类型、排除的项目（.md、node_modules、构建产物、静态资源）
3. **随机数函数清单**：根据用户项目的语言，从 `references/random-function-catalog.md` 中选取对应的 CSPRNG 和不安全函数列表
4. **执行原则**：
   - 分步执行：按文件逐一测试
   - 即时落盘：每完成一个文件立即将结果追加写入结果文件
   - SubAgent 隔离：每个文件一个独立 SubAgent，完成后释放
   - 仅记录发现：不提供修复方案
5. **审计阶段划分**：按风险优先级将文件分组为阶段（高风险目录 → 中风险 → 低风险 → UI/模板）
6. **SubAgent 标准化输入模板**：每个 SubAgent 收到的审计 prompt 格式
7. **输出文件说明**：结果文件路径、格式、追加方式
8. **完整文件清单**：列出所有待审计文件

### SubAgent 标准化 Prompt 模板

在测试计划中定义以下模板，后续每个文件的 SubAgent 都使用此格式：

```
你是一个白盒代码审计员。请审计以下文件中的随机数生成合规性。

**文件路径**: [完整路径]
**审计任务**:
1. 读取整个文件内容
2. 搜索所有随机数生成相关的函数调用或算法实现
3. 对每个命中位置，分析上下文判断是否处于安全敏感场景
4. 判断是否使用了正确等级的随机数函数

**安全随机数函数（应出现在安全场景）**:
[根据语言从 catalog 中选取]

**不安全随机数函数（在安全场景中为缺陷）**:
[根据语言从 catalog 中选取]

**判断规则**:
- 安全场景（加密/认证/Token/Salt/IV/Nonce/cnonce/协议随机数等）→ 必须用 CSPRNG
- 非安全场景（UI 动画/颜色/排序等）→ 可用非 CSPRNG

**重要约束 — 必须遵守**:
- **仅记录发现，绝对不提供修复方案或修复建议**。你的职责是审计和报告，不是修复代码。
- 不要输出"修复建议"、"应使用 xxx 替换"、"建议改为"、"推荐"等修复性内容
- 理由列仅说明为什么合规或为什么缺陷，不延伸到如何修改

**输出格式**:
若文件中有随机数生成位置，按表格输出：
| 行号 | 函数/模式 | 代码片段 | 上下文场景 | 合规判断 | 理由 |
|------|----------|---------|-----------|---------|------|
合规判断使用: ✅ 合规 / ❌ 缺陷 / ⚠️ 待确认
若文件中无随机数生成相关代码，输出: "无随机数生成代码"
最后给出小结: 共发现 N 处随机数使用，其中 M 处合规，K 处缺陷。
```

### 测试计划完成后

向用户展示测试计划的摘要（阶段划分、文件数量、预期流程），请用户确认后再进入 Phase 2。**测试计划文档本身不执行测试，仅作指导。**

---

## Phase 2: 执行审计

严格按测试计划执行。此阶段的核心复杂度在于管理大量文件的串行 SubAgent 调度。

### 2.1 初始化

在开始审计前，创建两个文件：

**结果文件** `[输出目录]/RandomSecurityAuditResults.md`：
```markdown
# 密码学安全随机数使用合规性审计结果

> 审计时间: YYYY-MM-DD
> 审计范围: [根目录]

## 汇总统计
（审计完成后填写）

## 详细审计结果
```

**进度文件** `[输出目录]/RandomSecurityAuditProgress.md`：
```markdown
# 审计进度日志

> 开始时间: YYYY-MM-DD

## 阶段 1: [阶段名] (N 文件)
- [ ] file1.ext
- [ ] file2.ext
...
```

### 2.2 逐文件审计循环

对每个阶段的每个文件**串行**执行：

```
FOR 每个文件:
  1. 主 Agent 创建 SubAgent(description="审计 [文件名]")
     - SubAgent 使用独立上下文
     - Prompt 来自测试计划中定义的模板
  2. SubAgent 读取文件 → 搜索模式 → 分析上下文 → 输出结构化结果
  3. 主 Agent 接收 SubAgent 结果
  4. **过滤修复建议**：若 SubAgent 结果包含修复性内容（"修复建议"、"建议改为"、"应使用 xxx 替换"），主 Agent 在写入前删除这些段落，仅保留审计表格和小结
  5. 若结果含随机数发现，立即追加写入结果文件
  6. 若无发现（"无随机数生成代码"），简短记录后继续
  7. 更新进度文件（将该文件标记为已完成）
  8. 继续下一个文件
```

**重要**：
- **必须等待当前 SubAgent 完成后才能启动下一个**，确保结果顺序落盘
- SubAgent 完成后其上下文自动释放，无需手动清理
- 若某个阶段文件较多（>15），可以将明显无风险的简单文件（如纯数据模型、枚举、常量定义）2~4 个合并在一个 SubAgent 中审计，在 prompt 中逐个指明文件路径即可

### 2.3 阶段间操作

每完成一个阶段后：
1. 在结果文件中写入**阶段小结**（阶段文件数、发现数、合规数、缺陷数）
2. 向用户简要报告阶段进度（如 "阶段 1 完成，4 文件，1 处发现，0 缺陷"）
3. 继续下一阶段

### 2.4 中断恢复

若审计因故中断：
- 查看 `RandomSecurityAuditProgress.md` 获取已完成文件列表
- 从下一个未完成的文件继续
- 结果追加到已有结果文件中

## Phase 3: 最终汇总

全部阶段完成后，更新结果文件的汇总统计部分：

```markdown
## 汇总统计

| 统计项 | 数量 |
|--------|------|
| 审计文件总数 | [N] |
| 发现随机数使用的文件数 | [N] |
| CSPRNG 合规使用（安全场景） | [N] |
| 非 CSPRNG 使用（非安全场景） | [N] |
| 缺陷（安全场景 + 非 CSPRNG） | [N] |
| Math.random() / 不安全函数使用 | [N] |
| 待确认 | [N] |

## 缺陷清单

[如有缺陷，逐一列出文件路径、行号、问题描述]

## 结论

[1-2 段总结性陈述]
```

向用户报告最终结论。重点说明：
1. 有无发现安全缺陷
2. 项目是否遵守了编码规范中的随机数要求
3. 架构约束是否得到遵守

## 安全敏感度判断规则

SubAgent 在分析每个随机数生成位置时，按以下规则判断是否需要 CSPRNG：

### 明确安全场景 → 必须用 CSPRNG
- 随机数用于生成密钥（AES/SM4/RSA/ECC）、IV、Nonce
- 随机数用于认证协议（Digest Auth cnonce、OAuth state 等）
- 随机数用于安全 Token（Session、CSRF、密码重置）
- 随机数用于通信协议防重放字段
- Salt 用于密码哈希
- 加密算法中的随机参数（如 ECC 随机数 k）

### 可能安全场景 → 建议用 CSPRNG
- 随机数用作唯一标识符（若可被猜测导致信息泄露）
- 随机端口、随机文件名等（视上下文）
- 验证码/OTP 生成

### 明确非安全场景 → 可用非 CSPRNG
- UI 动画效果、随机颜色
- 非安全关键的游戏机制
- 非安全敏感的数据洗牌/排序
- 随机展示内容（非安全关键）

### 字段名 vs 函数调用
- `random: string`（接口/类型定义中的字段名）→ 不算随机数生成
- `Math.random()`（实际函数调用）→ 算随机数生成

## 批处理策略

### 应逐文件独立审计的文件
- 安全模块（security/、crypto/、auth/）
- 网络通信模块（network/、http/）
- 服务/业务逻辑层（service/）

### 可合并批处理的文件
- 纯数据模型/Bean 定义（仅 interface/type/class 字段声明）
- 枚举/常量定义文件
- 纯 UI 组件（无业务逻辑的模板文件）
- 构建脚本和配置文件

合并时，在 SubAgent prompt 中逐个指明文件路径，要求每个文件输出一行结论。

## 参考资源

### 语言随机数函数目录
`references/random-function-catalog.md` — 覆盖 11 种语言/平台的 CSPRNG 与不安全随机数函数搜索模式。在撰写测试计划时，根据用户项目语言从该目录选取对应的函数列表。

### 使用方式
1. 审计前，根据用户项目语言查阅 catalog，选取对应的 CSPRNG 和不安全函数模式
2. 将这些模式填入 SubAgent 标准化 prompt 模板
3. SubAgent 在文件中搜索这些模式

## 注意事项

1. **不审计 .md 文件**：Markdown 文档不是源代码
2. **排除标准目录**：`node_modules/`、`vendor/`、`third_party/`、`build/`、`dist/`、`rawfile/`、`.git/`
3. **静态资源不算代码**：.css、图片、字体、资源 JSON（如 color.json、string.json）不审计
4. **不提供修复方案**：审计仅记录发现，修复由开发人员自行决定
5. **字段名不等于函数调用**：接口中的 `random: string` 字段声明不是随机数生成
6. **关注上下文而非孤立的函数名**：同一个函数名在不同上下文中安全意义不同（如 CRC32 中的 "random" 另当别论）
