# Skill: url-ip-extractor-v2

# URL/IP 地址提取测试执行 Skill (V2)

本 skill 用于从工程源码和配置文件中提取所有硬编码的 URL 地址和 IP 地址。采用分步执行、独立 subagent、即时落盘策略，避免上下文溢出。

---

## 触发时机

当用户提到"提取 URL"、"提取 IP 地址"、"提取网址"、"URL/IP 检视"、"网络地址审计"、"硬编码地址查找"等关键词时触发此 skill。

## 第一步：向用户询问/确认（必须执行，不可跳过）

**此阶段绝对不可跳过。** 在收集到所有必问信息之前，禁止开始测试。

**提问规范**：逐一提问，每次 1~2 个问题，等用户回答后再问下一批。

**第 1 轮（必问）**：
> 1. 检视根目录路径？（如 `entry/src/main/ets`、`src/`）
> 2. 结果文件输出路径？（默认 `<根目录>/TestPlan/URLTest/`）

等待用户回复后，继续第 2 轮。

**第 2 轮（必问）**：
> 3. 是否有需要排除的目录？（如 `node_modules`、`dist`、`.git`）
> 4. 是否有特别关注的文件类型或关注点？（如配置文件 `*.properties`、Mock 数据中的 URL/IP）

等待用户回复后，口头总结关键参数，确认无误后进入测试计划阶段。

根据用户确认的信息，撰写测试计划文档，包含：

### 测试计划结构

```markdown
# URL/IP 地址提取测试计划

> 创建时间: YYYY-MM-DD
> 检视根目录: <用户提供的路径>

## 一、测试范围
- 包含目录/文件列表
- 排除范围

## 二、分阶段执行策略
| 阶段 | 目录 | 文件数 | 执行方式 |
|------|------|--------|---------|
| 1 | <目录1> | N | subagent 批量扫描 |

## 三、关注重点
- 硬编码完整 URL（http://、https://、rtsp:// 等）
- 硬编码 IP 地址（非字段定义）
- 配置文件中的外部服务器地址
- Mock 数据中的 URL/IP（可选）

## 四、忽略内容
- 工程内部 API 路径（如 /api/v1/...）
- URL 拼接表达式（如 `${baseUrl}${path}`）
- 端口号常量
- IP 选择逻辑代码
- 密码/凭证
- IP 字段定义（如 ipv4_ip: string）

## 五、输出格式
- 结果文件路径
- Markdown 表格格式
```

## 第三步：执行测试

### 3.1 核心执行策略

- 将文件按目录/风险分组，串行启动 subagent
- 每个文件启动独立 subagent 扫描
- 主 agent 等待 subagent 完成后**过滤修复建议**（删除"修复建议"、"应改为"等段落）
- 立即落盘结果，结果追加写入
- 每完成一个阶段，向用户报告进度
- 继续下一阶段

```
主 Agent
  │
  ├──→ Subagent 1 (扫描文件 A) → 返回结果 → 落盘
  ├──→ Subagent 2 (扫描文件 B) → 返回结果 → 落盘（追加）
  ├──→ Subagent 3 (扫描文件 C) → 返回结果 → 落盘（追加）
  │
  等待本批完成...
  │
  ├──→ Subagent 4 (扫描文件 D) → 返回结果 → 落盘（追加）
  ...
```

### 3.2 Subagent 任务模板

```
【Subagent 任务】

请阅读文件 <完整文件路径>，提取其中硬编码的 URL 地址和 IP 地址。

关注内容：
1. 硬编码的完整 URL 字符串（http://...、https://...、rtsp://...、ws://... 等）
2. 硬编码的 IP 地址（IPv4 如 192.168.x.x、10.x.x.x；IPv6 如 fe80::、::1 等）
3. 配置文件中的服务器域名/地址
4. Mock 数据中的 URL/IP（测试数据、模拟设备地址）
5. 注释中的 URL/IP 地址
6. URL 路径中嵌入的 IP 地址

忽略内容：
- 工程内部 API 路径（如 /api/v1/、/SDCAPI/ 等，无 scheme + host 的相对路径）
- URL 拼接表达式（如 `${baseUrl}${path}`、模板字符串中的动态 URL）
- 端口号常量（如 const PORT = 8080）
- IP 字段定义（如 ipv4_ip: string、ip: string）
- 变量名（deviceIp、hostUrl 等）
- import 语句中的包引用 URL（如 import ... from 'https://...'）

重要约束 — 必须遵守：
- 仅记录发现，绝对不提供修复方案或修复建议
- 不要输出"修复建议"、"应改为"、"建议修改"等修复性内容

输出格式：
若文件中有发现，按以下表格逐行输出：
| 行号 | 内容 | 类型 | 上下文 |
|------|------|------|--------|
类型标注: URL常量 / IP常量 / 配置URL / Mock数据 / 注释URL

若文件中无发现，输出: "无发现"
最后给出小结: 共发现 N 处 (URL常量 X, IP常量 Y, 其他 Z)
```

## 执行注意事项

### 防止上下文溢出

1. **不累积结果**：subagent 结果立即落盘，不在主 agent 上下文中累积
2. **分批执行**：每批 4-6 个文件，等待完成后启动下一批
3. **结果追加**：后续结果追加到文件末尾，不覆盖

### 关注点分级

| 级别 | 类型 | 处理方式 |
|------|------|---------|
| **高关注** | 硬编码完整 URL/IP | 重点记录 |
| **中关注** | 配置文件服务器地址 | 记录 |
| **低关注** | Mock 数据 URL/IP | 可选记录 |
| **忽略** | API 路径、拼接表达式、端口、IP选择逻辑、密码 | 不记录 |

### 文件类型优先级

| 优先级 | 文件类型 | 原因 |
|--------|---------|------|
| 高 | *.properties、*.json、*.yaml、*.env | 配置文件，常含服务器地址 |
| 高 | *Config*.ts、*Config*.ets、*Constants* | 配置代码 |
| 中 | *.ts、*.ets、*.js、*.vue、*.kt、*.java | 源码文件 |
| 排除 | *.md、README*、node_modules、dist、build | 文档/第三方/构建产物 |

## 注意事项

1. **不执行修复**：仅记录发现，不提供修复建议。SubAgent 输出的修复性内容由主 Agent 过滤删除后再落盘
2. **不评估安全**：仅提取地址，不判断安全风险等级
3. **客观记录**：标注文件路径和行号，所有匹配到的 URL/IP 均如实报告
4. **用户主导**：每个阶段需用户确认后继续
5. **重点关注完整 URL**：带 scheme（http/https/rtsp/ws/ftp）的完整 URL 是最高优先级发现
6. **保留所有地址**：localhost、0.0.0.0、127.0.0.1、example.com 等地址正常提取，不作特殊排除
7. **上下文管理**：严格执行分步落盘，每文件完成后立即写入，避免上下文溢出
